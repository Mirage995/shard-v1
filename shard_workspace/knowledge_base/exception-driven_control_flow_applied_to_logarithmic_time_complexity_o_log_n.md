# exception-driven control flow applied to logarithmic time complexity o(log n) — SHARD Cheat Sheet

## Key Concepts
- **Exception-Driven Control Flow**: Using exceptions as a mechanism to control program flow rather than just error handling
- **O(log n) Complexity**: Algorithms that reduce problem size by a constant factor each iteration (e.g., binary search, tree traversal)
- **Early Exit via Exceptions**: Leveraging exceptions to break out of deeply nested recursive calls or iteration structures in logarithmic algorithms
- **Search Termination**: Using exceptions to signal "found" conditions in binary search trees or sorted arrays
- **Stack Unwinding**: Exploiting exception propagation to efficiently exit from log-depth recursion without explicit return value checking
- **Boundary Violations**: Treating out-of-bounds conditions in binary search as exceptional cases rather than conditional checks

## Pro & Contro
| Pro | Contro |
|-----|--------|
| Eliminates nested conditionals in recursive logarithmic algorithms | Exception handling overhead can negate performance benefits of O(log n) |
| Provides clean separation between "found" and "not found" paths | Makes code harder to reason about and debug |
| Simplifies early termination in deep recursion (log depth) | Violates principle of exceptions for exceptional cases only |
| Can reduce boilerplate in binary search implementations | Poor performance in languages with expensive exception mechanisms |
| Natural fit for "search until found" semantics | Breaks expected control flow patterns, reducing maintainability |

## Practical Example
```python
class FoundException(Exception):
    def __init__(self, index, value):
        self.index = index
        self.value = value

def binary_search_exception(arr, target, left=0, right=None):
    """Binary search using exception for early exit - O(log n)"""
    if right is None:
        right = len(arr) - 1
    
    try:
        if left > right:
            raise ValueError("Not found")
        
        mid = (left + right) // 2
        
        if arr[mid] == target:
            raise FoundException(mid, arr[mid])
        elif arr[mid] < target:
            binary_search_exception(arr, target, mid + 1, right)
        else:
            binary_search_exception(arr, target, left, mid - 1)
            
    except FoundException as e:
        raise  # Re-raise to unwind the log(n) depth stack
    
# Usage
try:
    binary_search_exception([1, 3, 5, 7, 9, 11, 13], 7)
except FoundException as e:
    print(f"Found at index {e.index}: {e.value}")
except ValueError:
    print("Not found")
```

## SHARD's Take
While exception-driven control flow can technically simplify early exits in O(log n) algorithms by eliminating return value propagation through logarithmic recursion depth, this approach is fundamentally a misuse of exception mechanisms. The performance overhead of exception handling often negates the theoretical elegance, and the resulting code violates the principle of least astonishment—exceptions should signal exceptional conditions, not routine control flow in predictable logarithmic search patterns. Use traditional conditionals for clarity and maintainability.
