# REST API design patterns — SHARD Cheat Sheet

## Key Concepts
- **Resource-Based URLs**: Structure endpoints around nouns (resources) rather than verbs, e.g., `/users` instead of `/getUsers`
- **HTTP Methods**: Use GET (read), POST (create), PUT/PATCH (update), DELETE (remove) to define operations on resources
- **Statelessness**: Each request contains all information needed; server maintains no client session state
- **HATEOAS**: Hypermedia as the Engine of Application State - responses include links to related resources
- **Versioning**: Manage API evolution through URL (`/v1/users`), headers, or query parameters
- **Pagination**: Handle large datasets using limit/offset or cursor-based pagination
- **Filtering & Sorting**: Allow clients to refine results via query parameters (`?status=active&sort=created_at`)
- **Idempotency**: Ensure PUT, DELETE, and GET operations produce same result when repeated
- **Error Handling**: Return consistent error responses with appropriate HTTP status codes and descriptive messages
- **Rate Limiting**: Protect API from abuse using throttling with headers like `X-RateLimit-Remaining`

## Pro & Contro
| Pro | Contro |
|-----|--------|
| Standardized approach using HTTP makes APIs intuitive and widely understood | Strict REST constraints can be overly rigid for complex business operations |
| Statelessness enables horizontal scaling and improved reliability | Multiple round trips may be needed for related data (N+1 problem) |
| Clear separation of concerns through resource-based design | Real-time features require workarounds (webhooks, long-polling) |
| Leverages existing HTTP infrastructure (caching, security, tooling) | HATEOAS is rarely fully implemented due to complexity |
| Language and platform agnostic | Versioning strategies can lead to maintenance burden |

## Practical Example

```javascript
// Express.js REST API with common patterns
const express = require('express');
const app = express();
app.use(express.json());

// Resource-based endpoint with pagination and filtering
app.get('/api/v1/users', (req, res) => {
  const { page = 1, limit = 10, status } = req.query;
  
  // Simulate filtering and pagination
  let users = mockDatabase.users;
  if (status) users = users.filter(u => u.status === status);
  
  const startIndex = (page - 1) * limit;
  const paginatedUsers = users.slice(startIndex, startIndex + limit);
  
  res.json({
    data: paginatedUsers,
    meta: {
      page: parseInt(page),
      limit: parseInt(limit),
      total: users.length
    },
    links: {
      self: `/api/v1/users?page=${page}&limit=${limit}`,
      next: startIndex + limit < users.length ? 
        `/api/v1/users?page=${parseInt(page) + 1}&limit=${limit}` : null
    }
  });
});

// Idempotent PUT for full resource update
app.put('/api/v1/users/:id', (req, res) => {
  const user = mockDatabase.users.find(u => u.id === req.params.id);
  if (!user) return res.status(404).json({ error: 'User not found' });
  
  Object.assign(user, req.body);
  res.json({ data: user });
});

// Proper error handling
app.use((err, req, res, next) => {
  res.status(err.status || 500).json({
    error: {
      message: err.message,
      code: err.code,
      timestamp: new Date().toISOString()
    }
  });
});
```

## SHARD's Take
REST API design patterns provide a battle-tested foundation for building scalable web services, but pragmatism should trump dogma. While adhering to core principles like resource-based URLs and proper HTTP method usage creates predictable, maintainable APIs, teams should adapt patterns to their specific needs rather than forcing every operation into pure REST constraints. The real value lies in consistency and developer experience—whether that means implementing full HATEOAS or choosing GraphQL for complex data requirements depends on your use case.
