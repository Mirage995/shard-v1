## Key Concepts
- **Catch Narrow Exceptions**: Target specific exception types rather than broad catches to avoid masking unexpected errors
- **Exception Hierarchy**: Python exceptions form a hierarchy; catching parent classes catches all children (e.g., `Exception` catches almost everything)
- **Bare except**: Using `except:` without specifying an exception type catches everything, including system exits and keyboard interrupts
- **Multiple except blocks**: Handle different exception types with specific recovery strategies
- **Exception chaining**: Use `raise ... from ...` to preserve original exception context when re-raising
- **EAFP principle**: "Easier to Ask for Forgiveness than Permission" - Python's idiomatic approach favoring try/except over pre-checks

## Pro & Contro
| Pro | Contra |
|-----|--------|
| Prevents hiding bugs by catching only expected errors | Requires more code and knowledge of exception hierarchy |
| Makes error handling explicit and self-documenting | Can lead to verbose code with many except blocks |
| Allows targeted recovery strategies for different failures | May miss edge cases if you don't anticipate all exceptions |
| Easier debugging when unexpected exceptions propagate | Overly narrow catches might let through related errors |
| Better logging and monitoring of specific error types | Requires maintenance when underlying libraries change exceptions |

## Practical Example
```python
# ❌ BAD: Too broad, catches everything including KeyboardInterrupt
try:
    user_data = json.loads(response.text)
    result = process_data(user_data['items'])
except:
    print("Something went wrong")

# ❌ BAD: Still too broad, masks programming errors
try:
    user_data = json.loads(response.text)
    result = process_data(user_data['items'])
except Exception as e:
    print(f"Error: {e}")

# ✅ GOOD: Catch specific, expected exceptions
try:
    user_data = json.loads(response.text)
    result = process_data(user_data['items'])
except json.JSONDecodeError as e:
    logger.error(f"Invalid JSON response: {e}")
    return None
except KeyError as e:
    logger.error(f"Missing expected field: {e}")
    return None
except ValueError as e:
    logger.error(f"Invalid data format: {e}")
    return None

# ✅ BEST: Narrow catches with exception chaining
def fetch_user_profile(user_id):
    try:
        response = requests.get(f"/api/users/{user_id}")
        response.raise_for_status()
        return response.json()
    except requests.HTTPError as e:
        raise UserNotFoundError(f"User {user_id} not found") from e
    except requests.RequestException as e:
        raise ServiceUnavailableError("API temporarily unavailable") from e
    except json.JSONDecodeError as e:
        raise DataCorruptionError("Invalid response format") from e
```

## SHARD's Take
Catching narrow exceptions is fundamental to writing robust Python code because it distinguishes between errors you anticipate and can handle versus bugs you need to fix. The practice forces developers to think critically about failure modes and implement appropriate recovery strategies, while allowing unexpected errors to propagate and surface during development. When combined with proper logging and exception chaining, narrow exception handling creates self-documenting code that's both maintainable and debuggable.
