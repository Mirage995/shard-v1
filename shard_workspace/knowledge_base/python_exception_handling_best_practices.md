# Python exception handling best practices — SHARD Cheat Sheet

## Key Concepts

- **EAFP (Easier to Ask Forgiveness than Permission)**: Attempt operations and handle exceptions rather than checking preconditions—the Pythonic way
- **Catch Narrow Exceptions**: Always catch specific exception types (e.g., `FileNotFoundError`) instead of broad catches like `except Exception`
- **Fail Fast Principle**: Raise exceptions immediately when errors occur to keep failures close to their root causes
- **Exception Boundaries**: Let low-level functions raise exceptions; catch them at application edges (API endpoints, main loops)
- **Exception Chaining**: Use `raise NewException from original` to preserve full error context and traceback history
- **Avoid Silent Failures**: Never catch exceptions without logging, re-raising, or handling meaningfully—silent failures hide bugs
- **Custom Domain Exceptions**: Create specific exception classes for domain errors while using built-ins when they fit
- **Dual Error Reporting**: Log technical details with tracebacks for developers; show friendly messages to users
- **Exceptions ≠ Control Flow**: Reserve exceptions for error conditions, not routine branching logic

## Pro & Contro

| Pro | Contro |
|-----|--------|
| EAFP reduces code duplication and eliminates race conditions from checking then acting | Overusing exceptions for control flow harms performance and readability |
| Exception chaining with `from` preserves complete debugging context across abstraction layers | Catching broad exceptions masks unrelated bugs and makes debugging harder |
| Failing fast keeps errors close to their source, simplifying root cause analysis | Silent failures create unpredictable systems and hide critical issues |
| Exception boundaries separate core logic from error presentation concerns | Mixing exception handling throughout code creates tangled, hard-to-maintain logic |
| Custom exceptions provide clear domain semantics and type-safe error handling | Over-engineering exception hierarchies adds unnecessary complexity |

```python
class PaymentProcessingError(Exception):
    """Raised when payment processing fails"""
    pass

def process_payment(amount, card_token):
    """Low-level function raises specific exceptions"""
    if amount <= 0:
        raise ValueError(f"Invalid amount: {amount}")
    
    try:
        # EAFP: attempt operation, handle failure
        response = payment_gateway.charge(card_token, amount)
        return response.transaction_id
    except NetworkError as e:
        # Exception chaining preserves original context
        raise PaymentProcessingError(
            f"Failed to process ${amount} payment"
        ) from e

def checkout_endpoint(request):
    """Application boundary catches and reports appropriately"""
    try:
        transaction_id = process_payment(
            request.amount, 
            request.card_token
        )
        return {"success": True, "transaction_id": transaction_id}
    
    except ValueError as e:
        # Narrow catch: handle specific, expected errors
        logger.warning(f"Invalid checkout data: {e}")
        return {"error": "Please check your payment details"}
    
    except PaymentProcessingError as e:
        # Log technical details for developers
        logger.error(f"Payment failed: {e}", exc_info=True)
        # Return friendly message for users
        return {"error": "Payment processing failed. Please try again."}
    
    # Note: Not catching Exception—let unexpected errors propagate
```

## SHARD's Take

Python's exception system achieves elegance through the EAFP philosophy and exception boundaries, but requires discipline to avoid anti-patterns like silent failures and broad catches. The most powerful yet underutilized feature is exception chaining with `from`, which preserves debugging context across abstraction layers—this should be standard practice whenever wrapping exceptions. The key insight is architectural: exceptions flow upward through layers, with low-level code raising them and high-level boundaries deciding how to translate technical failures into appropriate responses for different audiences.
