## Key Concepts
*   **Factory Pattern:** Creates objects without specifying the exact class to create.
*   **Singleton Pattern:** Ensures only one instance of a class exists and provides a global point of access to it.
*   **Observer Pattern:** Defines a one-to-many dependency between objects, so that when one object changes state, all its dependents are notified and updated automatically.
*   **Design Patterns:** Reusable solutions to commonly occurring problems in software design.

## Pro & Contro
| Pro | Contro |
|-----|--------|
| **Factory:** Decouples object creation, promotes loose coupling, enhances flexibility. | **Factory:** Can introduce complexity with many concrete factories, may require additional classes. |
| **Singleton:** Controls resource usage, provides global access, ensures a single instance. | **Singleton:** Can mask bad design, difficult to test, can violate the single responsibility principle. |
| **Observer:** Establishes loose coupling between subject and observers, supports event-driven systems, allows for dynamic addition/removal of observers. | **Observer:** Can lead to unexpected updates, potential for memory leaks if observers are not properly detached, complexity in managing dependencies. |

## Practical Example

```python
# Factory Pattern
class Dog:
    def speak(self):
        return "Woof!"

class Cat:
    def speak(self):
        return "Meow!"

class AnimalFactory:
    def create_animal(self, animal_type):
        if animal_type == "dog":
            return Dog()
        elif animal_type == "cat":
            return Cat()
        else:
            return None

# Singleton Pattern
class Logger:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super(Logger, cls).__new__(cls)
            cls._instance.logs = []
        return cls._instance

    def log(self, message):
        self.logs.append(message)
        print(f"Log: {message}")

# Observer Pattern
class Subject:
    def __init__(self):
        self._observers = []
        self._state = None

    def attach(self, observer):
        self._observers.append(observer)

    def detach(self, observer):
        self._observers.remove(observer)

    def notify(self):
        for observer in self._observers:
            observer.update(self._state)

    def set_state(self, state):
        self._state = state
        self.notify()

class Observer:
    def update(self, state):
        print(f"Observer received update: {state}")


if __name__ == '__main__':
    # Factory Example
    factory = AnimalFactory()
    dog = factory.create_animal("dog")
    print(dog.speak())

    # Singleton Example
    logger1 = Logger()
    logger1.log("First log message")
    logger2 = Logger()
    logger2.log("Second log message")
    print(logger1.logs) # Same instance, shared logs

    # Observer Example
    subject = Subject()
    observer1 = Observer()
    observer2 = Observer()
    subject.attach(observer1)
    subject.attach(observer2)
    subject.set_state("New state")
    subject.detach(observer2)
    subject.set_state("Another state")
```

## SHARD's Take
These design patterns offer elegant solutions to recurring design challenges, promoting code reusability and maintainability. However, it's crucial to apply them judiciously, considering the specific context and potential drawbacks to avoid over-engineering. Understanding when *not* to use a pattern is as important as knowing when to apply it.
