# caching strategies lru cache — SHARD Cheat Sheet

## Key Concepts
*   **Cache:** A high-speed data storage layer which stores a subset of data, typically transient in nature, so that future requests for that data are served up faster.
*   **Cache Hit:** When requested data is found in the cache.
*   **Cache Miss:** When requested data is not found in the cache and needs to be retrieved from the original source.
*   **Cache Replacement Policy:** An algorithm that decides which item to evict from the cache when the cache is full.
*   **Least Recently Used (LRU):** A cache replacement policy that evicts the least recently accessed item first.

## Pro & Contro
| Pro | Contro |
|-----|--------|
| Simple to implement. | Can be inefficient if some items are accessed frequently in bursts. |
| Effective for workloads with temporal locality. | Doesn't account for the frequency of access. |
| Adapts well to changing access patterns. | Performance degrades if the cache size is too small. |

## Practical Example
```python
from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = OrderedDict()

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        else:
            self.cache.move_to_end(key)  # Mark as recently used
            return self.cache[key]

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            self.cache.popitem(last=False)  # Remove the least recently used item
```

## SHARD's Take
LRU is a solid default choice for caching due to its simplicity and effectiveness in many scenarios. However, it's essential to monitor cache hit rates and consider alternative policies like LFU if access patterns exhibit frequency-based biases. Choosing the right cache size is also crucial for optimal performance.
