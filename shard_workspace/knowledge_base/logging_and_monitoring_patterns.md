# logging and monitoring patterns — SHARD Cheat Sheet

## Key Concepts
- **Structured Logging**: Emit logs in machine-parseable formats (JSON, key-value pairs) rather than plain text for easier querying and analysis
- **Log Levels**: Hierarchical severity classification (DEBUG, INFO, WARN, ERROR, FATAL) to filter noise and prioritize issues
- **Correlation IDs**: Unique identifiers propagated across service boundaries to trace requests through distributed systems
- **Centralized Log Aggregation**: Collecting logs from multiple sources into a single system (ELK, Splunk, Datadog) for unified analysis
- **Metrics vs Logs vs Traces**: Metrics are aggregated measurements, logs are discrete events, traces follow request flows—use appropriately
- **Health Checks**: Endpoints exposing system status (liveness, readiness) for orchestrators and load balancers
- **Alerting Thresholds**: Define meaningful boundaries that trigger notifications without causing alert fatigue
- **Sampling**: Capture subset of high-volume data (especially traces) to balance observability with performance costs
- **Retention Policies**: Balance storage costs with compliance and debugging needs through tiered data lifecycle management
- **Semantic Logging**: Log business events and domain concepts, not just technical operations

## Pro & Contra
| Pro | Contro |
|-----|--------|
| Enables rapid debugging and root cause analysis in production | Excessive logging degrades performance and increases storage costs |
| Provides visibility into distributed system behavior | Poorly structured logs create noise that obscures real issues |
| Supports compliance, auditing, and security forensics | Sensitive data in logs creates security and privacy risks |
| Metrics enable proactive alerting before user impact | Alert fatigue from misconfigured thresholds reduces effectiveness |
| Facilitates capacity planning and performance optimization | Complex observability stacks require specialized expertise to maintain |

## Practical Example
```python
import logging
import json
import uuid
from datetime import datetime

# Structured logging setup
class JSONFormatter(logging.Formatter):
    def format(self, record):
        log_data = {
            "timestamp": datetime.utcnow().isoformat(),
            "level": record.levelname,
            "message": record.getMessage(),
            "correlation_id": getattr(record, 'correlation_id', None),
            "service": "payment-service",
            "environment": "production"
        }
        if record.exc_info:
            log_data["exception"] = self.formatException(record.exc_info)
        return json.dumps(log_data)

logger = logging.getLogger(__name__)
handler = logging.StreamHandler()
handler.setFormatter(JSONFormatter())
logger.addHandler(handler)
logger.setLevel(logging.INFO)

# Usage with correlation ID
def process_payment(amount, user_id):
    correlation_id = str(uuid.uuid4())
    logger.info(
        f"Processing payment: amount={amount}, user_id={user_id}",
        extra={"correlation_id": correlation_id}
    )
    
    try:
        # Business logic here
        if amount > 10000:
            logger.warning(
                f"High-value transaction detected",
                extra={"correlation_id": correlation_id, "amount": amount}
            )
    except Exception as e:
        logger.error(
            f"Payment processing failed",
            extra={"correlation_id": correlation_id},
            exc_info=True
        )
        raise

# Output: {"timestamp": "2024-01-15T10:30:45.123456", "level": "INFO", 
#          "message": "Processing payment: amount=5000, user_id=42", 
#          "correlation_id": "a1b2c3d4-...", "service": "payment-service", 
#          "environment": "production"}
```

## SHARD's Take
Effective logging and monitoring is about signal-to-noise ratio—instrument what matters, not everything. Structured logging with correlation IDs is non-negotiable in distributed systems, but the real art lies in defining meaningful metrics and alerts that predict problems before they cascade. The best observability strategy treats logs, metrics, and traces as complementary tools, each optimized for their specific use case rather than trying to force one pattern to solve all problems.
