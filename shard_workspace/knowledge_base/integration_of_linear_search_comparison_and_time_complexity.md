# Integration of linear search comparison and time complexity -- SHARD Cheat Sheet

## Key Concepts
*   **Linear Search:** A simple search algorithm that sequentially checks each element in a list until a match is found or the entire list has been searched.
*   **Time Complexity:** A measure of the amount of time taken by an algorithm as a function of the input size.
*   **Worst-Case Time Complexity (O(n)):** The maximum time an algorithm could take to complete, occurring when the target element is at the end or not present.
*   **Average-Case Time Complexity (O(n)):** The average time an algorithm takes, assuming the target element is equally likely to be at any position.
*   **Best-Case Time Complexity (O(1)):** The minimum time an algorithm could take, occurring when the target element is the first element in the list.

## Pro & Contro
| Pro | Contro |
|-----|--------|
| Simple to implement | Inefficient for large datasets |
| Doesn't require sorted data | Time complexity increases linearly with the size of the input |
| Works well for small lists | Other algorithms (e.g., binary search) are much faster for larger lists |

## Practical Example
```python
def linear_search(list_data, target):
    for index, element in enumerate(list_data):
        if element == target:
            return index  # Return the index if found
    return -1  # Return -1 if not found

my_list = [5, 2, 9, 1, 5, 6]
target_value = 9
result = linear_search(my_list, target_value)

if result != -1:
    print(f"Element found at index: {result}")
else:
    print("Element not found in the list")
```

## SHARD's Take
Linear search is easy to understand and implement, making it suitable for small, unsorted datasets. However, its O(n) time complexity makes it impractical for large datasets where more efficient search algorithms like binary search (O(log n) - requires sorted data) or hash table lookups (O(1) average case) are preferable. Understanding its limitations is crucial for choosing the right search algorithm.
